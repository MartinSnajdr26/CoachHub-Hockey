# Database Migration Readiness — CoachHub Hockey

Date: 2026-07-20
Status: **Not migrating now.** This documents how close the codebase is to a
future move off SQLite and what to do when the pilot outgrows it.

## Current database configuration

- Engine: **SQLite**, file path from `.env` `DB_URL` (e.g. `sqlite:///coach/dev.db`),
  resolved to an absolute path by `_resolve_sqlite_url()` in `app.py`.
- ORM: SQLAlchemy 2.0.43 via Flask-SQLAlchemy 3.1.1; migrations via
  Flask-Migrate/Alembic (`migrations/`, single head, 16 revisions).
- Engine options (added in this pass): `pool_pre_ping=True`, and **for SQLite only**
  `connect_args={'timeout': 30}`. These are gated on the URL scheme so they are
  **not** applied to a future Postgres/MySQL URL.
- Session: scoped session; Flask-SQLAlchemy removes it on app-context teardown.

## SQLite-specific code found

Very little. The application is ORM-only and portable:

- **No** raw SQL, `PRAGMA`, `sqlite_master`, `json_extract`, `group_concat`,
  `AUTOINCREMENT`, or `sqlite3` imports in application code.
- JSON payloads are stored as `db.Text` (a JSON **string**), not a native JSON
  column — portable to any backend as-is.
- `datetime.strftime(...)` calls are Python-side formatting, not SQL.
- Only two SQLite-aware spots, both already isolated:
  - `_resolve_sqlite_url()` in `app.py` (only touches `sqlite:///` URLs; a
    non-sqlite URL is returned unchanged).
  - the sqlite-only `connect_args` in `SQLALCHEMY_ENGINE_OPTIONS` (guarded).

## Blockers for moving to PostgreSQL/MySQL

- **None structural.** Practical items to handle at migration time:
  - Timestamps are stored as **naive UTC** (`datetime.utcnow`). Keep the same
    convention; if using Postgres, prefer `TIMESTAMP WITHOUT TIME ZONE` to avoid
    implicit conversions. Do **not** switch columns to `timestamptz` blindly.
  - Boolean/int semantics: SQLite is loose about types; Postgres/MySQL are strict.
    Verify any place that compares e.g. a status string vs int after migration.
  - Case-sensitivity of `LIKE` differs (SQLite `LIKE` is case-insensitive for
    ASCII; Postgres `LIKE` is case-sensitive — use `ILIKE`). The audit-log
    `AuditEvent.event.like(prefix + '%')` filter (in `services/logging.py`) uses
    ASCII prefixes, so this is low risk, but re-test the audit log after migration.

## Migration risks

- Data type coercion during the SQLite → target copy (esp. datetimes, booleans, NULLs).
- Autoincrement/sequence reset: ensure target sequences continue past the max id.
- Alembic revisions were authored/tested on SQLite; a couple of ops (e.g.
  `ALTER TABLE` batch mode) are SQLite-driven. Re-run `flask db upgrade` against a
  **fresh empty** target DB (the Alembic chain builds the schema natively there),
  then load data — do not try to `ALTER` a copied SQLite file into Postgres.

## Recommended target for PythonAnywhere

- **MySQL** (bundled with paid PythonAnywhere plans, no extra hosting). It removes
  the single-writer file-lock problem that SQLite has under concurrency.
- PostgreSQL is also fine if hosted externally, but that adds a network hop and a
  managed service; for a small pilot MySQL-on-PythonAnywhere is the least friction.

## Proposed migration sequence

1. Provision the target DB (MySQL) on PythonAnywhere; note host/db/user.
2. Add driver to `requirements.txt` (`PyMySQL` for MySQL, or `psycopg[binary]` for PG).
3. Set `DB_URL` env var to the target (see below). Keep SQLite `.env` as a fallback for local dev.
4. Create schema on the target with Alembic: `flask db upgrade` (empty DB → native DDL).
5. Export current SQLite data and import into the target (small dataset → a short
   Python/ETL script iterating models is safest; preserves types via the ORM).
6. Run the app in a staging web app against the target; run the test suite.
7. Cut over `DB_URL` on the production web app; reload; verify; keep the SQLite file as a backup.

## Required environment variables (no real credentials here)

```
# MySQL example (fill in real values on the server; never commit them)
DB_URL=mysql+pymysql://USER:PASSWORD@HOST/DBNAME?charset=utf8mb4
# PostgreSQL example
# DB_URL=postgresql+psycopg://USER:PASSWORD@HOST:5432/DBNAME
```

`DATABASE_URL` is also accepted (app reads `DB_URL` first, then `DATABASE_URL`).

## Data backup & validation steps

- Backup: copy the SQLite file (`cp coach/<db>.db coach/<db>.db.bak-YYYYMMDD`) and
  keep the working web app running until validation passes.
- Validate after import: row counts per table match source; spot-check a team's
  players, attendance entries, calendar events, league cache JSON, and audit log;
  confirm login + dashboard render for at least one coach and one player key.

## Alembic / Flask-Migrate considerations

- Keep the single-head chain. Generate new revisions with
  `flask db migrate -m "..."` and review the autogenerated ops **before** upgrade
  (autogenerate can miss server-default/type nuances across backends).
- Do not rely on `db.create_all()` in production — Alembic is the source of truth.

## Rollback approach

- Cutover is a single env-var change + reload. Rollback = point `DB_URL` back at the
  SQLite file and reload. Because the SQLite file is untouched during the copy, it
  remains a valid rollback target. Keep it read-consistent by taking the backup at
  cutover and not writing to SQLite afterward.

## Parts to re-test after migration

- Login (coach + player keys), dashboard render, calendar CRUD (single + recurring),
  attendance save + export, pokladna, drills/lines save + PDF export, audit log
  (`LIKE`/prefix filter), league cache read + manual refresh, Týmuj import.
- Concurrency: two sessions writing at once (the whole point of leaving SQLite).
